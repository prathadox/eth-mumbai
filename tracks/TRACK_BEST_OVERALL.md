# Best Overall Project

Penguin Protocol is a complete, deployed, end-to-end private payroll system. It solves a real problem — on-chain salary exposure — with a stack that touches every meaningful layer of the Ethereum ecosystem: identity, custody, storage, ZK cryptography, and smart contracts.

---

## Why It Qualifies

Most hackathon projects integrate one or two sponsor tools and demo a single flow. This is a full protocol with five independent technical components that each solve a distinct part of the same problem, and which compose cleanly into a working product.

---

## The Full Stack, Layer by Layer

**Identity — ENS**

Employee onboarding doesn't use a database invite code or a custom registry contract. The backend calls `NameWrapper.setSubnodeRecord` on Sepolia and mints `alice.company.eth` directly to the employee's wallet. That subdomain is the authentication credential: SIWE verifies wallet ownership, the backend checks on-chain ownership, issues a JWT. Employee public keys and contract integrity hashes are written back as ENS text records — verifiable by anyone, requiring no custom infrastructure.

**Custody — BitGo**

Company treasury lives in a BitGo managed wallet. A scheduled cron job decrypts each employee's salary, evaluates the payment interval, and calls BitGo's SDK to move USDC into ShieldVault programmatically. The company never touches a raw private key. BitGo handles signing, fee management, and transaction broadcast. The handoff to the privacy layer is a standard ERC-20 transfer — the two systems don't need to know anything about each other.

**Document storage — Fileverse**

Employment contracts are encrypted with AES-256-GCM before upload. The AES key is wrapped with ECIES for both the company and the employee. The ciphertext goes to Fileverse, which pins it to IPFS. Fileverse sees an opaque blob. Decryption happens in the browser — the employee signs a deterministic message, the signature derives the AES key locally, the contract decrypts client-side. No server ever sees salary figures. No private key is ever stored.

**Privacy rails — ShieldVault + ZK**

ShieldVault on Base Sepolia maintains a MiMC Merkle tree. Deposits are commitments — `MiMC(amount, nonce, claim_pubkey, 0)` — no employee address on-chain. Withdrawals require a Barretenberg UltraHonk ZK proof generated by a Noir circuit that proves Merkle membership, nullifier validity, and stealth address derivation without revealing the underlying secret. USDC goes to a stealth address with no transaction history and no link to the employee's ENS name or primary wallet.

**The ZK circuit itself**

The Noir circuit implements MiMC over BN254 with no black-box operations — same constants as the Solidity verifier, hand-verified. The circuit covers six constraints: claim secret derivation, claim pubkey derivation, stealth address derivation, commitment construction, Merkle path verification, and nullifier derivation. The verifier contract was generated with `bb write_solidity_verifier` and deployed. Proofs are generated offline and served through the API.

## Technical Depth

The hardest part of this project isn't any single component — it's that every component has to agree on the same cryptographic constants. The MiMC round constants are hardcoded identically in:
- The Noir circuit (`main.nr`)
- The Solidity `ShieldVault.sol` MiMC library
- The TypeScript stealth data generation script (`generate-stealth-data.ts`)

A mismatch anywhere means the proof doesn't verify. Getting three independent implementations of the same hash function to agree — across Noir, Solidity, and TypeScript, over BN254 scalar field arithmetic — is the kind of constraint that eliminates most teams from attempting this construction at all.
