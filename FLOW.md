## Step 2: Invite Employee → Create ENS Subdomain

**File:** [`app/company/dashboard/page.tsx` → `handleInvite()`](penguin-protocol/penguin-protocol/app/company/dashboard/page.tsx#L131)
**API:** [`app/api/employees/invite/route.ts`](penguin-protocol/penguin-protocol/app/api/employees/invite/route.ts)

1. Company enters employee wallet address + ENS label (e.g. `alice`)
2. Company signs a wallet tx calling **ENS NameWrapper** `setSubnodeRecord()` — creates `alice.acme.eth` on Ethereum Sepolia
3. After tx confirms, calls `/api/employees/invite` → stores employee record in Supabase with status `pending`

```
wallet tx → ENS NameWrapper.setSubnodeRecord(acme.eth, "alice", employeeWallet)
                              ↓
           Supabase: employees { ens_name: "alice.acme.eth", status: "pending" }
```

## Step 3: Employee Claims ENS Subdomain

**Page:** [`app/employee/claim/page.tsx`](penguin-protocol/penguin-protocol/app/employee/claim/page.tsx)
**API:** [`app/api/employees/claim/route.ts`](penguin-protocol/penguin-protocol/app/api/employees/claim/route.ts)
**ENS verify:** [`app/api/employees/verify-ens/route.ts`](penguin-protocol/penguin-protocol/app/api/employees/verify-ens/route.ts)

1. Employee connects wallet, SIWE signs in
2. Server verifies on-chain that `alice.acme.eth` resolves to their wallet via [`lib/ens.ts`](penguin-protocol/penguin-protocol/lib/ens.ts)
3. Employee's **ECIES public key** is recovered from their SIWE signature: `ethers.SigningKey.recoverPublicKey(hash, sig)` — see [`app/api/auth/siwe/route.ts`](penguin-protocol/penguin-protocol/app/api/auth/siwe/route.ts#L17)
4. Public key stored in Supabase, status → `claimed`

```
SIWE sig → recover secp256k1 pubkey → store on employee record
ENS on-chain verify: getENSOwner("alice.acme.eth") === employeeWallet
```

## Step 4: Company Issues Encrypted Contract → Stored on Fileverse

**File:** [`app/company/dashboard/page.tsx` → `handleCreateContract()`](penguin-protocol/penguin-protocol/app/company/dashboard/page.tsx#L201)
**API:** [`app/api/contracts/create/route.ts`](penguin-protocol/penguin-protocol/app/api/contracts/create/route.ts)
**Encryption:** [`lib/encryption.ts`](penguin-protocol/penguin-protocol/lib/encryption.ts)
**Fileverse upload:** [`lib/fileverse.ts` → `uploadEncryptedContract()`](penguin-protocol/penguin-protocol/lib/fileverse.ts#L29)

1. Company fills in salary (1000 USDC) + interval (monthly) → clicks "Issue Encrypted Contract"
2. Server builds contract payload: `{ employeeEns, salary, interval, issuedAt, issuedBy }`
3. Contract is **ECIES-encrypted with both company and employee public keys** — only they can decrypt
4. Encrypted contract is uploaded to **Fileverse** (on-chain file registry + Pinata IPFS) → returns `fileId`
5. `keccak256(fileId)` stored as `doc_hash` in Supabase alongside the `fileverse_file_id`
6. Employee status → `active`

```
contractPayload
    → encryptContract(payload, companyPubKey, employeePubKey)   [lib/encryption.ts]
    → uploadEncryptedContract(encrypted, metadata)               [lib/fileverse.ts]
    → Fileverse Agent.create(markdownContent)
    → IPFS (Pinata) + on-chain file registry (Sepolia)
    → fileId stored in Supabase contracts table
```

## Step 5: Fund ShieldVault (Treasury → ZK Vault)

**Script:** [`scripts/treasury-to-vault.ts`](scripts/treasury-to-vault.ts)

1. Approves ShieldVault to spend MockUSDC
2. Calls `depositBatch(commitments[], amounts[])` — inserts 6 ZK commitments into on-chain Merkle tree (Base Sepolia)
3. Each commitment corresponds to a stealth address derived from an employee's secret
4. 6,000 USDC locked in vault, Merkle root recorded on-chain
5. Deposit record uploaded to **Fileverse** as audit trail — see [`scripts/treasury-to-vault.ts` step 5/5](scripts/treasury-to-vault.ts#L154)

```
COMMITMENTS[6] (one per employee stealth address)
    → ShieldVault.depositBatch(commitments, amounts)             [Base Sepolia]
    → Merkle root = 0x16724eb...
    → Fileverse: deposit record (tx hash, root, commitments)
```

**ZK commitments pre-generated by Noir circuit:**
[`circuits/`](circuits/) — UltraHonk / Barretenberg
[`scripts/generated-data/proofs/proof-index.json`](scripts/generated-data/proofs/proof-index.json) — maps proof keys → nullifier hashes → stealth addresses

## Step 6: Employee Decrypts Contract

**Page:** [`app/employee/contracts/page.tsx` → `handleDecrypt()`](penguin-protocol/penguin-protocol/app/employee/contracts/page.tsx#L103)

1. Employee clicks "Unlock with wallet" — signs deterministic message `"Penguin Protocol: decrypt key v1"`
2. The `r` component of the ECDSA signature is used as a symmetric decryption key (same key every time, derived from their private key, never sent to server)
3. Click "Decrypt" → shows salary: **1000 USDC**, interval: monthly

```
signMessage("Penguin Protocol: decrypt key v1")
    → ethers.Signature.from(sig).r   ← decryption key
    → decryptContractBrowser(encryptedPayload, r)
    → { salary: 1000, interval: "monthly", ... }
```

## Step 7: Employee Claims Salary via ZK Proof

**Page:** [`app/employee/contracts/page.tsx` → `handleClaim()`](penguin-protocol/penguin-protocol/app/employee/contracts/page.tsx#L122)
**API:** [`app/api/contracts/claim/route.ts`](penguin-protocol/penguin-protocol/app/api/contracts/claim/route.ts)
**Proof index:** [`scripts/generated-data/proofs/proof-index.json`](scripts/generated-data/proofs/proof-index.json)

1. Employee clicks "Claim" — frontend calls `POST /api/contracts/claim { key: "carol_stealth0" }`
2. Server loads pre-generated **UltraHonk ZK proof** from disk
3. Calls `ShieldVault.withdrawToStealth(proof, merkleRoot, nullifierHash, stealthAddress, amount)` on Base Sepolia using `SIGNER_PRIVATE_KEY`
4. Contract verifies the proof on-chain:
   - Proof says: *"I know a secret whose commitment is in the Merkle tree"*
   - No link revealed between which commitment = which employee
5. Nullifier recorded on-chain → double-claim blocked forever
6. 1000 USDC transferred to stealth address

```
proof (6336 bytes, UltraHonk)
    → ShieldVault.withdrawToStealth(proof, merkleRoot, nullifier, stealthAddr, 1000)
    → on-chain verify: is nullifier fresh? is Merkle root known? is proof valid?
    → USDC → stealthAddress (no on-chain link to employee identity)
    → nullifiers[nullifierHash] = true  ← cannot claim again
```

**Nullifier check script:** [`scripts/check-nullifiers.ts`](scripts/check-nullifiers.ts)

## Privacy Guarantee

| What's on-chain | What's hidden |
|---|---|
| 6000 USDC deposited to vault | Which commitment = which employee |
| N USDC withdrawn to some address | Which deposit → which withdrawal |
| Nullifier spent (anti-double-claim) | Employee identity, salary amount |
| Merkle root | Proof contents |

The only link between deposit and withdrawal exists inside the ZK proof — which is verified but never stored on-chain.
